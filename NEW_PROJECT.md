# Starting a New Project with Ralph

How to use the ralph loop to build a complete frontend application from an API reference.

## Prerequisites

- An AI coding tool installed: `claude`, `codex`, or `kiro-cli`
- `git` and `pnpm`
- An API reference URL (documentation page listing all operations)

## What to Copy

Copy these 4 files into your new empty project directory:

```bash
mkdir my-project && cd my-project
cp /path/to/ralph/{loop.sh,PROMPT_specs.md,PROMPT_plan.md,PROMPT_build.md} .
chmod +x loop.sh
git init
```

You do **not** copy `AGENTS.md`, `specs/`, or `IMPLEMENTATION_PLAN.md` — those are generated by the loop.

## Three Phases

```
┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│  Phase 1: SPECS          Phase 2: PLAN          Phase 3: BUILD       │
│                                                                      │
│  You provide:            Reads specs/,           Reads plan,          │
│  - API docs URL          analyzes gaps,          picks one task,      │
│  - Requirements          generates               implements,          │
│                          IMPLEMENTATION_PLAN.md   verifies, commits    │
│                                                                      │
│  Generates:              Generates:              Generates:           │
│  - specs/*.md            - IMPLEMENTATION_PLAN.md - src/* code        │
│  - AGENTS.md             - Task dependencies      - Tests             │
│                                                   - Git commits       │
│                                                   - AGENTS.md updates │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

## Step-by-Step

### 1. Configure your API URL

Open `PROMPT_specs.md` and replace the `API_URL` line:

```markdown
API_URL: https://docs.example.com/api/operations.html
```

If your API has a per-operation URL pattern, add that too. The agent will use these to fetch and understand every API operation.

### 2. Customize requirements (optional)

In `PROMPT_specs.md`, review the **Tech Stack** and **Core Requirements** sections. The defaults are:

- Vite + React + TypeScript (strict)
- TanStack Router + TanStack Query
- react-hook-form
- Cloudscape Design System
- MSW for API mocking
- Vitest + Playwright for testing
- react-i18next for internationalization

Modify these if your project uses a different stack.

### 3. Generate specs

```bash
./loop.sh --mode specs 3
```

This will:
- Fetch your API reference and extract every operation
- Group operations by domain
- Generate one spec file per domain in `specs/`
- Generate core infrastructure specs (routing, permissions, i18n, forms, testing)
- Generate a properly-structured `AGENTS.md` with your real stack and API doc URLs

**After it finishes:** review the generated `specs/` directory and `AGENTS.md`. Edit anything that's off — these are your source of truth.

### 4. Generate the implementation plan

```bash
./loop.sh --mode plan 3
```

This will:
- Read all specs
- Analyze dependencies between features
- Generate `IMPLEMENTATION_PLAN.md` with ordered, sized tasks
- Each task fits within a single AI context window

**After it finishes:** review `IMPLEMENTATION_PLAN.md`. Reorder tasks, adjust priorities, or split oversized tasks if needed.

### 5. Build

```bash
./loop.sh --mode build 200
```

This will loop autonomously:
1. Pick the highest-priority pending task
2. Investigate the codebase
3. Implement the task
4. Run backpressure checks (typecheck, lint, test)
5. Commit on success
6. Update the plan and progress log
7. Context destroyed, next iteration starts fresh

Set the iteration count high enough for your project. Each task takes one iteration. A project with 50 tasks needs at least 50 iterations (plus retries for blocked tasks).

### 6. Monitor

While the loop runs:
- **Terminal** — all agent output streams live
- **`IMPLEMENTATION_PLAN.md`** — check task statuses (done/pending/blocked)
- **`progress.txt`** — append-only log of each iteration
- **`git log --oneline`** — one commit per completed task
- **`AGENTS.md`** (bottom) — operational learnings Ralph discovered

### 7. Course correct

If something goes wrong:
- **Ctrl+C** to stop the loop at any time
- Edit specs, plan, or `AGENTS.md` as needed
- Restart the appropriate phase

| Problem | Fix |
|---------|-----|
| Specs are wrong | Edit `specs/*.md`, re-run `--mode plan` |
| Plan has bad ordering | Edit `IMPLEMENTATION_PLAN.md` directly |
| Ralph keeps failing on a task | Add guidance to `AGENTS.md` Operational Learnings |
| Ralph went off track | Delete `IMPLEMENTATION_PLAN.md`, re-run `--mode plan` |
| Need to change stack/approach | Edit specs + `AGENTS.md`, re-plan |

## Switching Tools

The loop is tool-agnostic. Switch at any time:

```bash
./loop.sh --tool claude --mode build 200
./loop.sh --tool codex --mode build 200
./loop.sh --tool kiro --mode build 200
```

All tools read the same `AGENTS.md`, specs, and plan files.

## Example: Full Run

```bash
# Setup
mkdir cognito-manager && cd cognito-manager
cp ~/ralph-template/{loop.sh,PROMPT_specs.md,PROMPT_plan.md,PROMPT_build.md} .
chmod +x loop.sh
git init && git add -A && git commit -m "initial ralph setup"

# Edit PROMPT_specs.md with your API URL...

# Generate everything and build
./loop.sh --mode specs 3       # ~5 min — generates specs + AGENTS.md
./loop.sh --mode plan 3        # ~5 min — generates implementation plan
./loop.sh --mode build 200     # hours — builds the entire project
```
