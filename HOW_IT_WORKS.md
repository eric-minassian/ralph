# Ralph Loop — How It Works

A tool-agnostic autonomous AI coding loop based on [Geoffrey Huntley's Ralph pattern](https://ghuntley.com/loop/).

Ralph runs a fresh AI agent instance in a loop. Each iteration picks one task, implements it, verifies it, commits it, and exits. The next iteration starts with clean context and picks up where the last one left off. Memory persists through files, not context windows.

## File Structure

```
ralph/
├── loop.sh                  # Bash loop — spawns fresh AI instances
├── AGENTS.md                # Operational guide (loaded by all AI tools each iteration)
├── PROMPT_plan.md           # Planning mode instructions
├── PROMPT_build.md          # Building mode instructions
├── specs/                   # Requirement specs (one per feature/JTBD)
│   └── TEMPLATE.md          # Copy this for each new spec
├── IMPLEMENTATION_PLAN.md   # Generated by plan mode, consumed by build mode
├── progress.txt             # Append-only log across iterations (gitignored)
├── archive/                 # Archived progress from previous runs (gitignored)
└── .gitignore
```

## Two Modes

### Plan Mode (`--mode plan`)

Reads specs and the current codebase, then generates `IMPLEMENTATION_PLAN.md` — a prioritized task list with dependencies, acceptance criteria, and file targets.

**No code is written. No commits are made.**

Run this first, or whenever the plan is stale or wrong.

### Build Mode (`--mode build`)

Reads the plan and picks the highest-priority pending task. Implements it, runs verification (typecheck, lint, test), commits on success, updates the plan, and logs progress.

**One task per iteration. Always.**

## The Loop

```
┌──────────────────────────────────────────────┐
│              loop.sh starts                  │
│                                              │
│  for each iteration (1..max):                │
│    ┌──────────────────────────────────────┐  │
│    │  Fresh AI instance spawned           │  │
│    │                                      │  │
│    │  Reads: AGENTS.md                    │  │
│    │         PROMPT_{mode}.md             │  │
│    │         IMPLEMENTATION_PLAN.md       │  │
│    │         progress.txt                 │  │
│    │         specs/*                      │  │
│    │         src/* (codebase)             │  │
│    │                                      │  │
│    │  Does work (plan or build)           │  │
│    │                                      │  │
│    │  Writes: IMPLEMENTATION_PLAN.md      │  │
│    │          progress.txt                │  │
│    │          AGENTS.md (learnings)       │  │
│    │          src/* (code changes)        │  │
│    │          git commit                  │  │
│    │                                      │  │
│    │  Context destroyed ←── this is key   │  │
│    └──────────────────────────────────────┘  │
│                                              │
│    if output contains <ralph>COMPLETE</ralph>│
│      → exit success                          │
│    else                                      │
│      → next iteration                        │
│                                              │
└─────────────────────────────────────────────┘
```

The fresh context per iteration is the core insight. LLM performance degrades as context fills. By resetting each iteration, every task gets the model's best work.

## Quick Start

### 1. Write your specs

Copy `specs/TEMPLATE.md` for each feature. Be specific about acceptance criteria — these become Ralph's definition of "done."

```bash
cp specs/TEMPLATE.md specs/user-auth.md
# edit specs/user-auth.md with your requirements
```

### 2. Configure backpressure

Edit `AGENTS.md` and replace the placeholder commands with your real build/test/lint commands. This is what prevents Ralph from committing broken code.

```markdown
## Verify (backpressure)

npm run typecheck
npm run lint
npm run test
```

### 3. Run planning

```bash
./loop.sh --mode plan 3
```

Review the generated `IMPLEMENTATION_PLAN.md`. Edit it if needed — Ralph treats it as mutable.

### 4. Run building

```bash
./loop.sh --mode build 20
```

Watch the output. If Ralph gets stuck, Ctrl+C out, adjust the plan or specs, and restart.

## Usage Reference

```bash
./loop.sh [options] [max_iterations]

Options:
  --tool <claude|codex|kiro>   AI tool to use (default: claude)
  --mode <plan|build>          Prompt mode (default: build)
  --model <model_id>           Model override (tool-specific)
  -h, --help                   Show help

Examples:
  ./loop.sh                              # claude, build, 10 iterations
  ./loop.sh --mode plan 3                # claude, plan, 3 iterations
  ./loop.sh --tool codex --mode build 20 # codex, build, 20 iterations
  ./loop.sh --tool kiro --mode build 15  # kiro, build, 15 iterations
```

## How Memory Works

Ralph has no persistent context between iterations. Instead, memory lives in files:

| File                     | Purpose                                         | Who writes it                                        |
| ------------------------ | ----------------------------------------------- | ---------------------------------------------------- |
| `IMPLEMENTATION_PLAN.md` | Task list, status, dependencies                 | Plan mode creates; build mode updates                |
| `progress.txt`           | Append-only log of what happened each iteration | Build mode appends                                   |
| `AGENTS.md`              | Operational learnings and gotchas               | Build mode adds entries when patterns are discovered |
| `git history`            | Code changes with commit messages               | Build mode commits                                   |

## Backpressure

Backpressure is what makes Ralph reliable. Before any commit, the agent must run verification commands (defined in `AGENTS.md`) and get them to pass. If verification fails 3 times, the task is marked `blocked` and Ralph moves on.

This creates a feedback loop:

```
implement → verify → fail → fix → verify → pass → commit
```

Without backpressure, Ralph will confidently commit broken code. Always define your verification commands.

## Tips

- **Start with small specs.** A 4-task plan is better than a 40-task plan for your first run.
- **Watch the first 2-3 iterations.** You'll learn what signals Ralph needs.
- **Keep AGENTS.md lean.** Start near-empty. Add entries only when Ralph hits a recurring problem. ~60 lines max.
- **Specs are immutable during build.** If requirements change, update specs yourself and re-run plan mode.
- **The plan is disposable.** If Ralph is going in the wrong direction, delete `IMPLEMENTATION_PLAN.md` and re-plan.
- **Run in a sandbox.** The loop uses `--dangerously-skip-permissions` (or equivalent). Use Docker, a VM, or a disposable branch.
